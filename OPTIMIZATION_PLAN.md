# 番茄小说阅读器（Fanqie Reader）优化计划（按 Gemini 意见修订）

> **修订说明**：本版本不再使用“功能堆砌式愿望清单”的写法，而改为**贴合当前代码库现状、以可用性与可维护性优先、带明确优先级和验收标准**的执行计划。
>
> **核心原则**：
> 1. 先修主链路，再做体验增强。
> 2. 先消除当前代码/配置/接口的不一致，再引入新依赖。
> 3. 已经完成或已有基础能力的部分，不重复作为高优先级目标。
> 4. 每个优化项都必须说明：**目标、涉及文件、实施方式、验收标准**。

---

## 一、Gemini 评审结论（整理后）

根据 Gemini 的意见，原优化计划存在以下问题：

1. **过于理想化，缺少与现有项目状态的对应关系**
   - 原文把 PWA、虚拟滚动、阅读器大改、数据库迁移等全部并列，但没有区分哪些是“当前痛点”，哪些只是“以后可以做”。
   - 容易导致 AI 或开发者优先去做炫技项，而不是先补齐真实可用性问题。

2. **部分目标与当前代码库重复或冲突**
   - 当前前端已经有主题系统、现代化 UI、搜索骨架屏、卡片化布局、内部 API 模式等基础能力。
   - 原文仍将这些作为“从零开始”的目标描述，容易重复建设。

3. **缺少“主链路验收”视角**
   - 真正决定项目是否可用的，是这条链路是否稳定：
     **搜索 → 加入下载队列 → 任务推进 → 小说详情 → 章节阅读 → 导出/词云/封面**。
   - 原文没有把主链路放在最高优先级。

4. **缺少明确的延后项和非目标**
   - 像 PWA、全面视觉重做、数据库大迁移，不应在主链路尚未完全稳定前优先推进。

因此，本计划改为 **P0 / P1 / P2 / P3** 四级路线图，并补充“当前已具备基础”和“暂缓项”。

---

## 二、当前项目基线（避免重复优化）

以下能力当前项目**已经具备基础**，后续优化应在此基础上迭代，而不是重做：

### 1. 前端已有基础
- Vue 3 + Vite + Pinia + Vue Router + Element Plus + ECharts。
- 首页/搜索/详情/任务/设置等页面已经完成现代化 UI 改造。
- 已有明暗主题切换能力（`frontend/src/composables/useTheme.ts`）。
- 搜索页已具备：
  - 卡片式布局
  - 骨架屏
  - 排序与筛选
  - 自动加载更多
  - 批量下载
- 章节阅读页已具备：
  - 字体大小调整
  - 阅读模式切换
  - 上一章/下一章导航基础

### 2. 后端已有基础
- Flask + SQLAlchemy + Socket.IO + Celery 架构已成型。
- `backend/app.py` 已提供搜索、小说、章节、任务、统计、下载、封面等核心 API。
- 已支持内部 API 模式。
- 已有下载任务、搜索缓存、封面缓存、词云等基础数据模型。

### 3. 最近已修复的关键问题（应视为当前基线的一部分）
- 内部 API 模式下 WebSocket 与认证逻辑冲突。
- 任务页进度百分比显示错误。
- 阅读页章节前后跳转只依赖前 10 章的问题。
- 小说详情页路由监听参数错误。
- PostgreSQL 下注册用户序列未同步导致注册失败。
- 缺失词云时返回 200 而不是 404 的接口语义问题。
- 本地开发时 Vite 代理目标硬编码为 `backend:5000` 的问题。

## 三、优化优先级总览

| 优先级 | 目标 | 是否应立即做 |
|---|---|---|
| P0 | 主链路可用性与一致性 | 是 |
| P1 | 阅读体验与页面性能优化 | 是，但应建立在 P0 稳定后 |
| P2 | 架构治理、测试、可维护性提升 | 是，逐步推进 |
| P3 | PWA / 高级视觉增强 / 可选工程化 | 否，后置 |

---

## 四、P0：主链路可用性与一致性（最高优先级）

> 目标：保证“可运行、可到达、可维护”的最小闭环稳定。

### P0-1. 补齐主链路冒烟测试
**目标**：把真实使用路径变成可重复验证的检查清单，而不是靠手点。

**涉及文件**：
- `backend/app.py`
- `backend/test_api.py`
- 建议新增：`backend/tests/` 或 `scripts/smoke/`

**实施方式**：
- 固化以下检查路径：
  1. `/healthz`
  2. `/api/system/info`
  3. `/api/search`
  4. `/api/novels`（列表 / 详情 / 章节）
  5. `/api/tasks/list`
  6. `/api/novels/<id>/cover`
  7. `/api/stats/wordcloud/<id>`
- 对“应返回 404 的资源缺失场景”与“应返回 200 的有效资源场景”分别覆盖。
- 若下载链路依赖外部网络，至少保留一套“无网络/本地 broker 条件下”的可控测试。

**验收标准**：
- 本地可一键执行冒烟测试。
- 测试输出能明确区分：代码错误 / 配置错误 / 外部网络错误。

---

### P0-2. 前后端接口契约统一
**目标**：避免前端字段名、状态码、错误语义和后端不一致。

**重点问题**：
- 字段命名不一致（如 `total_chapters` / `total_chapters_source` 一类问题）。
- 资源不存在却返回 200 的情况。
- 某些接口返回 `{ error: ... }` 但 HTTP 码不标准。

**涉及文件**：
- `backend/app.py`
- `frontend/src/api.ts`
- `frontend/src/store/index.ts`
- 相关视图组件

**实施方式**：
- 为所有核心接口建立“返回契约清单”。
- 前端类型定义只保留一套真源，不接受“猜字段”。
- 统一约定：
  - 资源不存在 → 404
  - 参数错误 → 400
  - 业务冲突（如重复任务）→ 409
  - 服务器异常 → 500

**验收标准**：
- `frontend/src/api.ts` 中主要接口不再需要为历史兼容做重复转换。
- 页面错误提示基于真实 HTTP 语义即可工作。

---

### P0-3. 运行环境一致性与依赖健康
**目标**：解决“代码没问题，但本地跑不起来/换环境就坏”的问题。

**已知风险**：
- Python 全局环境中存在异常包残留警告。
- 后端测试脚本容易受到代理环境污染。
- 前端构建依赖此前存在缺失安装问题。

**涉及文件**：
- `backend/requirements.txt`
- `frontend/package.json`
- `frontend/vite.config.ts`
- `.env.example`
- `README.md`

**实施方式**：
- 补齐并校验开发依赖与测试依赖。
- 明确区分：
  - 本地开发默认值
  - Docker 环境默认值
  - 外部服务依赖（Redis / Celery / 第三方 API）
- 文档中明确：哪些功能必须依赖外部网络，哪些可本地自测。

**验收标准**：
- 前端 `type-check` / `build` 可直接运行。
- 后端至少 `compileall`、基础测试、最小接口验证能直接运行。

---

### P0-4. 任务系统一致性
**目标**：确保任务页展示、WebSocket 更新、后端状态语义完全一致。

**涉及文件**：
- `backend/app.py`
- `backend/tasks.py`
- `frontend/src/api.ts`
- `frontend/src/store/index.ts`
- `frontend/src/views/TasksView.vue`

**实施方式**：
- 统一任务状态含义：`PENDING / DOWNLOADING / PROCESSING / COMPLETED / FAILED / TERMINATED`。
- 统一进度取值范围（0-100）。
- 明确“删除任务”的 websocket 事件结构。
- 区分“任务失败”和“文件尚未生成导致下载失败”的展示文案。

**验收标准**：
- 任务页在内部 API 模式与认证模式下都能正常显示状态。
- 进度条、状态文案、操作按钮行为一致。

---

## 五、P1：阅读体验与页面性能优化

> 目标：在主链路稳定后，优先提升用户最能感知到的体验。

### P1-1. 阅读器增强（基于现有 `ChapterView.vue` 继续迭代）
**目标**：从“能读”提升到“好读”。

**涉及文件**：
- `frontend/src/views/ChapterView.vue`
- 可新增 `reader` store 或 composable

**实施方式**：
- 在现有字号 / 阅读模式基础上，补充：
  - 背景主题：浅色 / 深色 / 羊皮纸 / 护眼绿 / OLED Dark
  - 行距、段距调节
  - 阅读设置面板（抽屉或弹层）
- 增加阅读进度记忆：
  - 记录小说 ID、章节 ID、滚动位置
  - 详情页提供“继续阅读”入口
- 增加快捷键：上一章 / 下一章 / 向下翻页

**验收标准**：
- 关闭页面后再次进入，可恢复到最近阅读位置。
- 设置项与主题切换不会互相打架。

---

### P1-2. 小说详情页章节目录性能优化
**目标**：解决章节数很大时的渲染压力。

**注意**：
- 该项不是立即引入，需先确认真实数据量是否足够大。
- 如果大多数小说章节数不高，可优先保留分页方案。

**涉及文件**：
- `frontend/src/views/NovelDetailView.vue`

**实施方式**：
- 优先方案：继续使用分页，但支持更合理的每页大小与快速跳页。
- 若确有数千章场景，再引入虚拟列表（`el-table-v2` 或虚拟滚动方案）。

**验收标准**：
- 500+ 章节时详情页滚动和切换不卡顿。
- 不为了“技术炫技”无条件引入重型虚拟化改造。

---

### P1-3. 骨架屏补齐与空状态统一
**目标**：消除不同页面加载体验不一致的问题。

**当前状态**：
- 搜索页已有骨架屏基础。
- 详情页、任务页、Dashboard 部分区域仍可继续统一。

**涉及文件**：
- `frontend/src/views/SearchView.vue`
- `frontend/src/views/NovelDetailView.vue`
- `frontend/src/views/TasksView.vue`
- `frontend/src/views/DashboardView.vue`

**实施方式**：
- 为详情页元数据、任务列表、Dashboard 卡片增加统一骨架策略。
- 统一空状态文案、错误状态文案、重试按钮位置。

**验收标准**：
- 页面加载时不再出现明显闪烁、白块或状态跳变。

---

## 六、P2：架构治理与可维护性提升

### P2-1. 全局错误处理收口
**目标**：减少各页面重复 `try...catch + ElMessage` 逻辑，但不牺牲业务提示准确性。

**涉及文件**：
- `frontend/src/api.ts`
- `frontend/src/store/index.ts`

**实施方式**：
- 将“通用网络错误”收敛到 Axios 响应拦截器。
- 将“业务动作反馈”（如“已加入任务队列”）继续保留在组件层。
- 避免把所有错误都做成一刀切的全局弹窗。

**验收标准**：
- 重复错误处理代码减少。
- 页面成功提示与失败提示职责清晰。

---

### P2-2. 拆分过重的 Store 与 View 逻辑
**目标**：降低维护成本，避免 `store/index.ts` 与超长页面继续膨胀。

**涉及文件**：
- `frontend/src/store/index.ts`
- `frontend/src/views/*`

**实施方式**：
- 按域拆分 store：`auth`、`novels`、`tasks`、`stats`、`reader`。
- 将与页面无关的逻辑继续抽到 composables。
- 保持“视图负责展示，store/composable 负责状态与副作用”。

**验收标准**：
- 新增功能时，不需要继续把所有逻辑塞进单个 store 文件。

---

### P2-3. 数据库迁移机制标准化
**目标**：停止继续在运行时手写 schema 修补逻辑扩散。

**涉及文件**：
- `backend/app.py`
- 数据模型文件
- 迁移目录（待新增）

**实施方式**：
- 引入 `Flask-Migrate` / Alembic。
- 将未来结构调整通过 migration 管理。
- 当前运行时补丁逻辑保留兜底，但不继续扩张。

**验收标准**：
- 新字段 / 新索引的变更不再依赖人工 SQL 与运行时 patch。

---

### P2-4. Celery 重试与任务可观测性
**目标**：提升任务执行的鲁棒性与问题定位能力。

**涉及文件**：
- `backend/tasks.py`
- `backend/celery_init.py`

**实施方式**：
- 对可重试异常引入指数退避重试。
- 区分：网络错误、上游返回异常、文件写入失败、分析失败。
- 日志中保留 task id、novel id、关键阶段标记。

**验收标准**：
- 同类下载失败不再只能靠人工反复点击“重新下载”恢复。
- 从日志能快速定位失败阶段。

---

### P2-5. 自动化测试体系补齐
**目标**：给未来重构留安全网。

**建议优先级**：
1. API 冒烟测试
2. Store / composable 级测试
3. 下载器单元测试（Mock 外部请求）
4. 关键页面集成测试

**涉及文件**：
- `backend/tests/`
- `frontend/src/**/__tests__/`
- 或 Playwright / Vitest 配置

**验收标准**：
- 关键改动后，至少能自动回归：
  - 搜索
  - 任务列表
  - 小说详情
  - 章节阅读
  - 注册登录

---

## 七、P3：后置优化（不是当前首要任务）

### P3-1. PWA 支持
**结论**：可以做，但不应先做。

**原因**：
- 当前项目首先要保证主链路与下载能力稳定。
- PWA 的收益主要在“安装体验”和“弱网壳层缓存”，不是当前第一痛点。

**建议**：
- 放在 P2 之后。
- 前提是：路由、资源缓存、离线提示策略都已经整理清楚。

---

### P3-2. 高级视觉与动画增强
**结论**：只做增量，不做二次大改版。

**建议**：
- 保留现有玻璃态与主题体系。
- 只补：焦点态、悬停态、动效减弱策略、阅读模式对比度优化。
- 不建议再次大规模重写整体 UI。

---

### P3-3. 状态持久化插件化
**结论**：可选项，不是必须项。

**说明**：
- `pinia-plugin-persistedstate` 可提升体验，但当前已有 `useTheme` 与本地存储方案。
- 应在确认状态项持续增多时再引入，避免为插件而插件。

---

## 八、明确暂缓 / 非目标项

以下内容**当前不建议作为主线任务**：

1. **数据库整体迁移（如迁往 MongoDB）**
   - 除非用户明确提出，不纳入本计划主线。

2. **全面 UI 重做**
   - 当前已具备较完整的视觉体系，继续大改风险高、收益低。

3. **为所有列表强行引入虚拟滚动**
   - 需基于真实性能瓶颈，而不是先上技术方案。

4. **在未稳定前大量引入新依赖**
   - 每引入一个插件，都意味着新的维护面与升级风险。

---

## 九、推荐实施顺序（修订版）

### Phase 1：主链路稳定化（必须先做）
- API 冒烟测试固化
- 前后端接口契约统一
- 任务系统状态与错误语义统一
- 运行环境与文档对齐

**完成标志**：
- 搜索、任务、详情、章节、封面、词云、注册登录都能稳定验证。

### Phase 2：阅读与详情体验增强
- 阅读设置面板
- 阅读进度记忆
- 快捷键支持
- 详情页加载体验补齐

**完成标志**：
- 阅读页从“能用”升级到“好用”，且不引入新的状态混乱。

### Phase 3：架构治理
- 全局错误收口
- Store/composable 拆分
- Flask-Migrate
- Celery 重试与日志分层
- 自动化测试体系

**完成标志**：
- 新功能迭代时不再明显增加屎山程度。

### Phase 4：后置增强
- PWA
- 视觉细节优化
- 按需持久化插件
- 构建产物进一步切分优化

**完成标志**：
- 在不影响主链路稳定性的前提下，继续打磨产品完成度。

---

## 十、执行要求（给开发者 / AI 编程助手）

1. **每次只处理一个明确子目标**，不要一轮同时引入多种新依赖和大重构。
2. **修改前先确认当前文件是否已有同类实现**，避免重复造轮子。
3. **所有优化都必须带验证步骤**：至少说明如何构建、如何冒烟、如何回归。
4. **如果某项功能依赖外部上游服务**，必须在文档或提交说明中标出，不可把外部失败误判为代码失败。
5. **优先减少不一致、隐式行为和历史残留代码**，而不是单纯新增功能。

---

## 十一、一句话版结论

> 这个项目下一步最重要的，不是继续“加功能”，而是把现有的搜索、任务、阅读、导出、统计链路彻底做稳；在此基础上，再做阅读体验增强和架构治理，最后才考虑 PWA 与更重的工程化扩展。
